这个文件夹是在演示 **如何把 Biomni 本身暴露成一个 MCP server，并从外部客户端测试调用**。目录里只有两个代码文件：

* `run_mcp_server.py`
* `test_mcp_server.py` ([GitHub][1])

它们的关系很直接：

* `run_mcp_server.py` 负责**启动 Biomni MCP 服务端**
* `test_mcp_server.py` 负责**作为 MCP 客户端连接并测试某个工具** ([GitHub][2])

---

## 1. `run_mcp_server.py`

这个文件非常短，作用是把 Biomni agent 暴露成一个 **stdio 传输的 MCP server**。代码流程是：

1. 导入 `A1`
2. 创建 agent：`agent = A1()`
3. 调用 `agent.create_mcp_server(tool_modules=["biomni.tool.database"])`
4. 在主程序中打印启动信息并执行 `mcp.run(transport="stdio")` ([GitHub][2])

### 它实现的核心功能

它把 `biomni.tool.database` 这个模块中的工具，打包成一个可对外提供的 MCP server。也就是说，这个例子不是把整个 Biomni 所有工具都暴露出去，而是**只选择数据库工具模块**作为对外服务能力。([GitHub][2])

### 这个脚本的定位

它是一个最小 server 入口文件，主要演示三件事：

* Biomni 的 `A1` agent 可以创建 MCP server
* 可以按模块选择要暴露的工具
* 服务通过 `stdio` 方式运行，适合被本地 MCP client 子进程拉起和通信 ([GitHub][2])

---

## 2. `test_mcp_server.py`

这个文件是一个 **异步测试客户端**，用来验证上面的 MCP server 是否真的能工作。它使用了 MCP Python 客户端相关组件：

* `ClientSession`
* `StdioServerParameters`
* `stdio_client` ([GitHub][3])

### 它的默认测试目标

脚本顶部预设：

* `TOOL_TO_TEST = "query_uniprot"`
* `TEST_ARGS = {"prompt": "Find information about human insulin protein"}` ([GitHub][3])

也就是默认拿 Biomni 暴露出来的 `query_uniprot` 工具做一次测试，查询人胰岛素蛋白信息。([GitHub][3])

### 它还给了几个备选测试示例

代码里还注释了几组替代配置，例如：

* `blast_sequence`
* `region_to_ccre_screen`
* `query_emdb` ([GitHub][3])

这说明作者想把它做成一个通用的“单工具测试脚本”：你只要改顶部变量，就能切换测试别的 Biomni MCP 工具。([GitHub][3])

### 它的主要执行流程

`test_single_tool()` 做的事情比较完整：

1. 定位当前目录下的 `run_mcp_server.py`
2. 用 `python run_mcp_server.py` 作为子进程启动 MCP server
3. 复制当前环境变量传给子进程
4. 用 `stdio_client(...)` 建立连接
5. 创建 `ClientSession`
6. `initialize()`
7. `list_tools()` 列出服务端所有工具
8. 查找目标工具名是否存在
9. 打印工具描述和输入 schema
10. 调用 `session.call_tool(TOOL_TO_TEST, TEST_ARGS)`
11. 打印原始结果，并尽量把 JSON 结果解析成更易读的格式 ([GitHub][3])

所以它不仅是“调用一下看看”，还顺带实现了：

* 工具发现
* 参数 schema 展示
* 参数调试提示
* JSON 结果格式化
* 错误捕获与诊断 ([GitHub][3])

### 它的错误处理功能

这个脚本对调试很友好。比如：

* 如果目标工具不存在，会列出所有可用工具
* 如果调用失败，并且异常里像是 validation error，会提示你检查 `TEST_ARGS` 是否匹配 schema
* 如果结果不是 JSON，就只展示前 500 个字符 ([GitHub][3])

这说明它不仅是 demo，也是一份实际可用的本地联调脚本。([GitHub][3])

---

## 3. 这两个文件合起来演示的完整流程

整个文件夹其实在展示一条完整的 **“Biomni 作为 MCP 服务被外部调用”** 的链路：

1. 用 `run_mcp_server.py` 把 Biomni 的某个工具模块包装成 MCP server
2. 用 `test_mcp_server.py` 作为独立客户端，通过 stdio 连接这个 server
3. 枚举工具、检查 schema、发送参数、接收结果 ([GitHub][2])

和你前面看的 `add_mcp_server` 示例相比，这个目录展示的是反方向：

* `add_mcp_server`：**把外部 MCP server 接入 Biomni**
* `expose_biomni_server`：**把 Biomni 自己暴露成 MCP server 给外部用** ([GitHub][2])

---

## 4. 一句话总结

* `run_mcp_server.py`：把 `biomni.tool.database` 模块里的工具暴露成一个基于 stdio 的 Biomni MCP server。([GitHub][2])
* `test_mcp_server.py`：作为 MCP 客户端启动并连接这个 server，列出工具、检查输入 schema，并测试调用指定工具。([GitHub][3])
