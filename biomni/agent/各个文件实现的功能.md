这个 `biomni/agent` 文件夹是 Biomni 的 **agent 层**：里面放的是代理入口、任务型 agent、小型 LLM 包装器，以及一个 ReAct 风格 agent 的实现片段。目录当前包含 `__init__.py`、`a1.py`、`env_collection.py`、`function_generator.py`、`qa_llm.py`、`react.py` 这 6 个文件。([GitHub][1])

先给你一个整体判断：

* `a1.py` 是这里最核心的文件，定义了主代理 `A1`，负责加载配置、准备数据目录、下载 data lake / benchmark、初始化 LLM、工具注册、工具检索和 know-how 文档。([GitHub][2])
* `function_generator.py` 和 `qa_llm.py` 都是比较轻量的专用 agent / 包装器。前者负责把任务描述生成成 Python 代码，后者负责直接问答和结构化结果整理。([GitHub][3])
* `env_collection.py` 更像一个“从论文里抽取可泛化计算任务”的 agent。([GitHub][4])
* `react.py` 是一个 ReAct 风格 agent 的实现片段，强调工具调用、规划和超时包装，但从当前抓到的代码看，这个文件像是部分实现或被压缩/截断后的版本。([GitHub][5])
* `__init__.py` 只是包导出入口，把 `A1` 暴露出来。([GitHub][6])

下面按文件分别说。

## 1. `__init__.py`

这个文件只有一句：

```python
from biomni.agent.a1 import A1
```

它的作用很简单，就是把 `A1` 作为 `biomni.agent` 包的对外主入口。这样外部可以直接从包里导入 `A1`，而不必写更长的模块路径。([GitHub][6])

## 2. `a1.py`

这是这个目录里最重要的文件，实现了 Biomni 的主 agent 类 `A1`。代码里能看到它：

* 定义了 `AgentState`，包含 `messages` 和 `next_step`，说明它内部按消息状态驱动。([GitHub][2])
* 在初始化时读取默认配置 `default_config`，并支持覆盖 `path`、`llm`、`source`、`use_tool_retriever`、`timeout_seconds`、`base_url`、`api_key`、`commercial_mode` 等参数。([GitHub][2])
* 根据 `commercial_mode` 决定加载 `env_desc` 还是 `env_desc_cm`，也就是区分 academic/all datasets 与 commercial-only datasets。([GitHub][2])
* 会在本地创建 `biomni_data/benchmark` 和 `biomni_data/data_lake` 目录，并调用 `check_and_download_s3_files(...)` 从 `https://biomni-release.s3.amazonaws.com` 下载缺失的数据湖和 benchmark 文件。([GitHub][2])
* 初始化主 LLM：`self.llm = get_llm(...)`。([GitHub][2])
* 读取模块到 API 的映射：`module2api = read_module2api()`。([GitHub][2])
* 当 `use_tool_retriever=True` 时，会创建 `ToolRegistry(module2api)` 和 `ToolRetriever()`，说明它支持“工具注册 + 检索式工具选择”。([GitHub][2])
* 初始化 `KnowHowLoader()`，并在商业模式下过滤 know-how 文档。([GitHub][2])

所以 `A1` 的核心功能可以概括为：

**Biomni 的统一代理入口，负责环境准备、模型准备、工具准备、知识准备，然后把这些能力整合成一个可执行的科研 agent。** ([GitHub][2])

### `a1.py` 里几个特别重要的方法

`add_tool(self, api)`
这个方法支持把用户自定义函数动态加入到 agent 的工具系统里。它会：

* 读取函数源码 `inspect.getsource(api)`
* 用 `function_to_api_schema(...)` 自动生成 schema
* 补全 `name`、`description`、`required_parameters`
* 写入 `ToolRegistry`
* 更新 `module2api`
* 把原始函数放到 `_custom_functions` 和 `builtins._biomni_custom_functions` 里
* 最后重新 `configure()` agent。([GitHub][2])

这说明 `A1` 不只是用内置工具，还支持**运行时扩展自定义工具**。([GitHub][2])

`add_mcp(...)`
从注释能看出，它支持从配置文件动态注册 MCP server 工具，把 MCP 工具包装成 agent 可调用函数。虽然我们没有展开全部实现，但 docstring 已明确这个职责。([GitHub][2])

### 对 `a1.py` 的定位

如果把整个 Biomni 想成一个系统，`a1.py` 就是最像“总控台”的文件。
它不是某个单独任务 agent，而是把：

* LLM
* data lake
* benchmark
* tool registry
* know-how documents
* MCP tools
* custom tools

这些全部装配起来。([GitHub][2])

## 3. `env_collection.py`

这个文件定义了 `PaperTaskExtractor(base_agent)`。它的用途在类注释里写得很清楚：
**从学术论文中提取可泛化、可自动化的计算任务/实验任务。** ([GitHub][4])

它的特点有：

* 使用 `RecursiveCharacterTextSplitter`，说明会把论文拆成 chunk 处理。([GitHub][4])
* `configure()` 里定义了一段很长的 chunk analysis prompt，要求模型只提取“极其常见、标准、可代码实现、输入输出清晰”的计算任务，并且识别常用数据库和软件包。([GitHub][4])

这说明这个文件的目标不是回答论文内容本身，而是把论文里的方法学抽象成“可以复用为工具/函数的任务类型”。([GitHub][4])

从系统角度看，它像是一个 **方法学挖掘 agent** 或 **任务归纳 agent**。
比较适合做这类事情：

* 读一批论文
* 提炼出共性的计算流程
* 为后续工具开发或 benchmark 设计提供任务候选。 ([GitHub][4])

有一个值得注意的小点：它从 `biomni.agent.base_agent` 导入 `base_agent`，但当前 `agent` 目录文件列表里没有单独看到 `base_agent.py`。这说明要么该类在别处，要么这部分代码存在版本不一致/目录不完整的情况。这个是我基于当前目录内容做的判断。([GitHub][1])

## 4. `function_generator.py`

这个文件定义了一个简化版 `base_agent`，以及 `FunctionGenerator(base_agent)`。它的职责很明确：

**根据任务描述生成可执行的 Python 代码脚本。** ([GitHub][3])

它做了几件事：

* `base_agent` 里包装了 `get_llm(...)`，保存 `tools`、`llm`、`cheap_llm` 等基础属性。([GitHub][3])
* `FunctionGenerator.configure()` 定义了系统提示词，要求模型：

  * 只输出 Python 代码
  * 尽量放进单个 fenced code block
  * 适当加注释和 docstring
  * 有 `main()` 和 `if __name__ == '__main__'`
  * 尽量少依赖外部包
  * 优先利用公共仓库代码。([GitHub][3])
* `_generate_code()` 调用 LLM 生成代码。([GitHub][3])
* `_generate_script_filename()` 根据任务描述自动生成安全的 `.py` 文件名。([GitHub][3])
* `go()` 返回 `(script_filename, results)`，其中 `results` 是抽取出的代码块。([GitHub][3])

所以，这个文件本质上是 Biomni 里的 **代码生成 agent**。
适合从“任务描述”直接得到“脚本草稿”。([GitHub][3])

## 5. `qa_llm.py`

这个文件更小，是一个非常轻量的问答包装器 `qa_llm`。它的功能分两部分：

第一部分是最普通的 LLM 问答：

* `__init__()` 初始化 LLM；
* 如果 `lab_bench_reproduce=True`，会加一个 prompt modifier，要求回答生物学多选题时“只返回正确选项字母并 step by step 思考”；否则不加。([GitHub][7])
* `go(input)` 会把输入发给模型，记录 log，并返回原始回答。([GitHub][7])

第二部分是结果结构化：

* `result_formatting(output_class, task_intention)` 会构造一个 `ChatPromptTemplate`
* 让同一个 LLM 基于 `with_structured_output(output_class)` 从历史消息中抽取并格式化结果。([GitHub][7])

所以它的定位是：

**轻量问答 agent + 结果格式化器**。
它不像 `A1` 那样管工具和数据，也不像 `FunctionGenerator` 那样产代码，而是适合快速回答问题，再把回答整成结构化对象。([GitHub][7])

## 6. `react.py`

这个文件从当前抓到的内容看，是一个 **ReAct 风格 agent** 的实现片段，强调：

* 对工具函数增加超时保护 `create_timed_func(...)`
* 把已有工具包装成带超时限制的 `wrapped_tools`
* 动态 `add_tool(...)`
* `configure(...)` 里根据不同模式切换提示词，比如：

  * `react_code_search=True` 时，只允许用 `run_python_repl` 和 `search_google`
  * 普通模式下是“helpful biologist and expert geneticist”
  * `plan=True` 时，会要求先写 `Research Plan and Status`，持续记录高层计划、已完成步骤和结果。([GitHub][5])

所以，哪怕当前文件只抓到部分内容，也能看出它实现的是：

**一个有工具调用、规划、可反思/可搜索模式切换的 ReAct 代理。** ([GitHub][5])

不过这里我得明确一点：
当前 `react.py` 抓到的内容看起来像中间片段，不像完整文件开头，说明这个文件要么内容较长但解析展示不完整，要么 raw 抓取时压成了单行片段。基于目前看到的代码，我能可靠说明它的“功能方向”，但不能保证把每个方法都完整还原。([GitHub][5])

## 这些文件之间怎么分工

把这个目录合起来看，大致是这样的：

* `__init__.py`：包入口，只导出主 agent。([GitHub][6])
* `a1.py`：主控 agent，负责环境、工具、数据、知识、MCP、自定义工具接入。([GitHub][2])
* `env_collection.py`：从论文中抽取通用计算任务。([GitHub][4])
* `function_generator.py`：把任务描述变成 Python 代码。([GitHub][3])
* `qa_llm.py`：做轻量问答和结构化输出。([GitHub][7])
* `react.py`：做带工具调用和规划能力的 ReAct 代理。([GitHub][5])

所以这不是一组“平级的业务脚本”，而更像一套 agent 组件库：

* `A1` 负责总装配
* 其他几个文件负责特定代理范式或特定任务。 ([GitHub][2])

## 一句话总结

`biomni/agent` 目录实现的是 Biomni 的代理层：
**`a1.py` 是主代理与系统装配入口，`env_collection.py` 做论文任务抽取，`function_generator.py` 做代码生成，`qa_llm.py` 做轻量问答与结果结构化，`react.py` 提供带规划和工具调用的 ReAct 风格代理，`__init__.py` 则把 `A1` 暴露为包入口。** ([GitHub][6])

你接下来最值得看的，是 `a1.py` 里 `configure()` 和 `add_mcp()` 的完整实现；那两部分最能看出 Biomni 主 agent 的真正运行流程。
