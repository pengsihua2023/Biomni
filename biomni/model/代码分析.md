这个 `retriever.py` 实现的是一个很轻量的 **资源检索器**，核心类只有一个：`ToolRetriever`。它的主要功能不是向量检索，也不是 BM25，而是把“用户问题 + 候选资源列表”一起交给 LLM，让模型从中挑出更相关的 **工具、数据集、软件库、know-how 文档**。换句话说，它实现的是一种 **prompt-based resource retrieval**。([GitHub][1])

## 整体结构

这个文件很短，主要包含：

* 导入：`contextlib`、`re`、`HumanMessage`、`ChatOpenAI`
* 一个类：`ToolRetriever`
* 这个类的 3 个主要方法：

  * `prompt_based_retrieval(...)`
  * `_format_resources_for_prompt(...)`
  * `_parse_llm_response(...)` ([GitHub][1])

从代码看，`ToolRetriever.__init__()` 本身什么也没做，只是一个空初始化，所以真正的逻辑几乎都集中在 `prompt_based_retrieval` 及两个辅助函数里。([GitHub][1])

## 1. 这个类的核心用途

`ToolRetriever` 的目标是：
给定一个用户查询 `query`，以及一个 `resources` 字典，从里面挑出最相关的资源。代码注释明确要求 `resources` 里可以有这些键：

* `tools`
* `data_lake`
* `libraries`
* `know_how`

而返回值也是同样结构的字典，只不过每个类别里只保留“被认为相关”的那些条目。([GitHub][1])

所以它更像一个 **资源筛选器**，而不是资源生成器。它不产生新工具，也不执行工具，只负责“先把候选集缩小”。([GitHub][1])

## 2. `prompt_based_retrieval(...)` 做了什么

这是整个文件的主函数。它的输入是：

* `query: str`：用户问题
* `resources: dict`：候选资源池
* `llm=None`：可选的 LLM 实例；如果没传，就自己创建一个 `ChatOpenAI(model="gpt-4o")`。([GitHub][1])

它的处理流程可以拆成 5 步。

### 第一步：把候选资源整理成 prompt 文本

它会先构造一个 prompt，把用户问题放进去，然后依次加入：

* `AVAILABLE TOOLS`
* `AVAILABLE DATA LAKE ITEMS`
* `AVAILABLE SOFTWARE LIBRARIES`
* 如果有的话，再加 `AVAILABLE KNOW-HOW DOCUMENTS` ([GitHub][1])

这里用到的格式化函数是 `_format_resources_for_prompt(...)`，它会把每个候选资源都变成带编号的文本，例如：

`0. resource_name: description`

这样模型后面就可以只返回编号。([GitHub][1])

### 第二步：给模型明确的输出格式要求

代码要求模型按固定格式只返回索引列表，例如：

* `TOOLS: [0, 3, 5]`
* `DATA_LAKE: [1, 2]`
* `LIBRARIES: [0, 4]`
* 如果存在 know-how，再加 `KNOW_HOW: [...]` ([GitHub][1])

这一步很关键，因为后面解析阶段不是做语义理解，而是直接用正则表达式抽取这些方括号中的数字。也就是说，这段代码对模型输出格式有明显依赖。([GitHub][1])

### 第三步：在 prompt 里加入偏好规则

这个 retrieval prompt 不是中性的，它带有很强的策略倾向。代码里明确告诉模型：

* 要“宽松一些”选择，不要漏掉潜在有用资源
* 一般查询时，要优先包含数据库工具
* 要包含所有文献搜索工具
* 如果是湿实验/序列类问题，一定要包含分子生物学工具
* know-how 文档要选相关 protocol / best practice / troubleshooting
* 对数据库工具和分子生物学工具，拿不准时宁可多选也不要漏掉。([GitHub][1])

这意味着它不是单纯“最相似检索”，而是一个 **带人工偏好的 LLM 路由器**。([GitHub][1])

### 第四步：调用 LLM

如果外部没传 `llm`，它会默认创建 `ChatOpenAI(model="gpt-4o")`。
然后：

* 如果这个 `llm` 有 `.invoke(...)` 方法，就按 LangChain 风格传入 `[HumanMessage(content=prompt)]`
* 否则就把它当成普通可调用对象 `llm(prompt)` 使用。([GitHub][1])

这说明它做了一点兼容设计，既适配 LangChain 风格模型，也尽量兼容别的简单接口。([GitHub][1])

### 第五步：解析输出并筛选资源

模型返回后，代码会调用 `_parse_llm_response(...)` 提取各类别的索引列表，然后按这些索引去原始 `resources` 字典里取对应元素，组装成 `selected_resources` 返回。
它对每类资源都做了索引边界判断：只有 `i < len(...)` 才会真正取值，所以即使模型输出了越界索引，也不会直接报错。([GitHub][1])

## 3. `_format_resources_for_prompt(...)` 的作用

这个辅助函数负责把候选资源变成适合塞进 prompt 的编号文本。它支持 3 种输入格式：

* 如果资源是 `dict`，就取 `name` 和 `description`
* 如果资源是 `str`，就直接编号输出
* 否则尝试从对象属性里取 `.name` 和 `.description`，取不到就转成字符串。([GitHub][1])

这说明这个检索器对上游资源格式比较宽容，能接受：

* 注册表里的工具字典
* 简单字符串列表
* 带属性的对象列表。([GitHub][1])

如果某个类别没有任何资源，它会返回 `"None available"`，这样 prompt 中该部分也仍然是完整的。([GitHub][1])

## 4. `_parse_llm_response(...)` 的作用

这个函数负责把模型返回文本解析成内部可用的索引字典。它做了两层兼容。

第一层是输入类型兼容：
如果 `response` 是一个 list，它会把其中的文本块拼接起来；这主要是兼容 Responses API 风格的 content block 列表。
如果不是字符串，也会强制转成字符串。([GitHub][1])

第二层是字段解析：
它用正则分别匹配：

* `TOOLS: [...]`
* `DATA_LAKE: [...]`
* `LIBRARIES: [...]`
* `KNOW[-_]HOW: [...]` ([GitHub][1])

然后把里面逗号分隔的数字转成整数列表，放进：

```python
{"tools": [], "data_lake": [], "libraries": [], "know_how": []}
```

这个字典里。([GitHub][1])

这里还有一个细节：每次字符串转整数时都包在 `contextlib.suppress(ValueError)` 里，所以如果某一类输出格式坏掉，不会整个函数报错，只会让那一类保持空列表。([GitHub][1])

## 5. 这个实现的设计特点

这个文件实现很简单，但设计意图比较明确。

第一，它是 **LLM 驱动检索**，不是 embedding 驱动检索。
整个流程没有向量化、没有相似度计算、没有索引库，只是把候选项全文列给模型，让模型自己判断。([GitHub][1])

第二，它是 **多类别联合筛选**。
不是只选工具，还会同时考虑：

* tools
* data lake
* libraries
* know-how documents ([GitHub][1])

这说明它更像给一个科研 agent 做“资源路由前置层”，而不是单纯的 tool selector。([GitHub][1])

第三，它带有 **领域偏置**。
prompt 里明确强调 biomedical、database tools、literature tools、molecular biology tools、protocols 等，这个 retriever 是明显为生物医学研究代理写的。([GitHub][1])

## 6. 这个实现的优点

从工程角度看，这个文件的优点是：

* 很轻量，几乎零依赖复杂检索基础设施
* 能同时处理多种资源类型
* 对输入资源格式兼容性不错
* 对模型输出和格式错误有一定容错
* 很容易通过 prompt 调整检索策略。([GitHub][1])

## 7. 这个实现的局限

它也有几个明显限制。

第一，**扩展性一般**。
因为它把所有候选资源都直接塞进一个 prompt，如果工具、数据集、库太多，prompt 会很长，成本和延迟都会上升。([GitHub][1])

第二，**结果稳定性依赖模型输出格式**。
虽然有正则解析，但如果模型不按要求输出 `TOOLS: [...]` 这种格式，解析结果可能为空。([GitHub][1])

第三，**没有真正的排序分数**。
它只返回“选中哪些资源”，不返回每个资源的相关度分值，也没有 top-k 排名逻辑。([GitHub][1])

第四，**策略偏向很强**。
例如“总是优先 database tools”“包含所有 literature search tools”“湿实验问题总要包含 molecular biology tools”，这会让召回更高，但也可能带来过多冗余。这个是设计选择，不一定是缺陷，但会影响检索精度。([GitHub][1])

## 一句话总结

这个 `retriever.py` 实现的是一个 **基于 LLM 提示词的资源筛选器**：它把用户问题和候选的工具/数据/库/know-how 文档一起交给模型，让模型返回相关项的编号，再由代码解析编号并取出对应资源。它适合 Biomni 这种生物医学 agent 系统做前置资源路由，但不是传统意义上的向量检索器。([GitHub][1])
