这个 `biomni/task` 文件夹目前只有 4 个文件：`__init__.py`、`base_task.py`、`hle.py`、`lab_bench.py`。从实现上看，它们组成了 Biomni 的一个很轻量的 **任务定义层**：先用 `base_task.py` 定义统一任务接口，再用 `hle.py` 和 `lab_bench.py` 各自封装具体 benchmark 的数据读取、prompt 构造、迭代取样、评估和输出格式。目录页直接列出了这 4 个文件。([GitHub][1])

先说整体关系：

* `base_task.py` 是抽象基类风格的接口模板。
* `hle.py` 实现了一个 Humanity’s Last Exam 风格的生物/医学多选任务类。
* `lab_bench.py` 实现了 Lab Bench 数据集的多选任务类。
* `__init__.py` 目前是空文件，主要作用就是把这个目录标记为 Python 包。([GitHub][2])

### `__init__.py`

这个文件当前没有内容，raw 文件返回 0 行。它本身不实现业务逻辑，主要就是让 `biomni/task` 成为一个可导入的 Python 包。因为它没有导出语句，所以这里没有额外的封装层或公共入口逻辑。([GitHub][2])

### `base_task.py`

这个文件定义了一个非常简洁的 `base_task` 类，包含这些方法：

* `__init__`
* `get_example`
* `get_iterator`
* `evaluate`
* `output_class`
* `get_prompt_from_input` ([GitHub][3])

前五个方法基本都是空实现 `pass`，说明它是一个“接口模板”而不是可直接使用的任务类。它要求子类自己实现：如何取样本、如何遍历样本、如何评估输出、以及输出应该符合什么结构。唯一有默认实现的方法是 `get_prompt_from_input(self, input)`，它直接调用 `self.get_example(input)["prompt"]`，也就是把“根据样本索引取 prompt”统一成一个通用入口。([GitHub][3])

所以 `base_task.py` 的功能可以概括为：
**定义任务类的统一最小接口，让不同 benchmark 可以用相同方式取题、评估和声明结构化输出类型。** 这是一种很典型的轻量任务抽象层。([GitHub][3])

### `hle.py`

这个文件实现了 `humanity_last_exam(base_task)` 类，面向 Human­ity’s Last Exam 里生物/医学类别的多选题。代码一开始会校验 `category` 和 `answer_type` 参数是否合法；支持的 `category` 包括 `Other`、`Humanities/Social Science`、`Math`、`Physics`、`Computer Science/AI`、`Biology/Medicine`、`Chemistry`、`Engineering`，支持的 `answer_type` 包括 `exactMatch` 和 `multipleChoice`。([GitHub][4])

不过从实际实现看，它虽然接收 `category` 和 `answer_type` 参数，但默认会读取固定文件 `path + "/hle/test_sampled_biology_medicine.parquet"`，然后再在 DataFrame 内部进一步筛选 `df["category"] == self.dataset` 且 `df["answer_type"] == "multipleChoice"`。这说明当前这个实现实际聚焦在 **生物/医学多选题** 上，而不是一个完全通用的 HLE 全科加载器。([GitHub][4])

`hle.py` 还定义了两个局部辅助函数：

* `extract_options(question)`：从题目字符串里 `"Answer Choices:"` 后面解析出 A、B、C... 的选项文本。
* `extract_question(question)`：把题干和 `"Answer Choices:"` 部分切开，只保留题干。([GitHub][4])

但从当前代码能看到的实际赋值逻辑里，`df["question_text"] = df.question`，也就是最终直接把原始 `question` 列拿来做 `self.query`；解析选项的函数虽然写了，但在后面的核心流程里没有明显真正用来重组 prompt。这说明它可能是作者中途保留的辅助代码，或者用于更早/更完整版本的预处理。这个判断是基于当前文件内容做的推断。([GitHub][4])

这个类的主要字段包括：

* `self.query`：题目文本
* `self.answer`：正确选项字母，来自 `answer` 的首字符
* `self.prompt`：模板 `"Question: {question}"`。([GitHub][4])

对应的方法实现如下：

* `get_example(index=None)`：返回一个字典，至少包含 `prompt` 和 `answer`；如果不传索引，就随机抽一道题。
* `get_iterator()`：顺序遍历所有样本，逐个 `yield self.get_example(i)`。
* `evaluate(response)`：把模型输出与真值比较，返回 `accuracy`、`coverage`、`refrain_ratio`、`precision`。
* `output_class()`：返回一个 Pydantic 模型 `MultipleChoiceOutput`，其中只有一个字段 `choice`，要求输出类似 `"A"` 这样的选项字母。([GitHub][4])

这里有一个实现细节值得注意：`evaluate()` 里使用了 `self.refrain_label` 来计算 `coverage` 和 `refrain_ratio`，但在 `hle.py` 当前代码片段里并没有看到 `self.refrain_label` 被定义。相比之下，`lab_bench.py` 明确设置了 `self.refrain_label`。这说明 `hle.py` 的 `evaluate()` 可能存在一个实现缺口，或者依赖了未展示出来的其他上下文；就当前文件本身来看，这里有潜在 bug 风险。([GitHub][4])

所以，`hle.py` 的核心功能是：
**把 HLE 生物/医学多选题 benchmark 封装成一个任务对象，支持随机取题、批量遍历、选项字母评估和结构化输出声明。** ([GitHub][4])

### `lab_bench.py`

这个文件实现了 `lab_bench(base_task)` 类，用来封装 Lab Bench 风格的生物学多选 benchmark。初始化参数 `dataset` 只允许 `"DbQA"` 或 `"SeqQA"`，然后从 `path + "/" + dataset + "/train-00000-of-00001_test.parquet"` 读取对应数据。([GitHub][5])

它的 prompt 模板比 `hle.py` 更完整、更强调自动评测格式。模板明确告诉模型：

* 这是一个 biology multiple choice question
* 必须回答正确选项字母
* 必须把字母放在 `[ANSWER]` 和 `[/ANSWER]` 标签之间
* 必须严格使用这个格式，便于自动解析。([GitHub][5])

`lab_bench.py` 的一个关键设计是：它会把原始数据里的 `distractors`、`ideal` 正确答案，再加上一项 `"Insufficient information to answer the question."` 一起打乱，形成最终选项列表。随后它会生成：

* `options_letters`：带 `A.`、`B.`、`C.` 前缀的可展示选项文本
* `letter_answer`：正确答案对应的字母
* `letter_refrain`：代表“信息不足，拒答”选项对应的字母。([GitHub][5])

这说明 `lab_bench.py` 和 `hle.py` 最大的区别之一，是它原生支持 **refrain / abstain** 机制，也就是允许模型在信息不足时选“无法回答”。这也解释了为什么它的 `evaluate()` 中 `coverage`、`refrain_ratio`、`precision` 这些指标是完整闭环的。([GitHub][5])

这个类主要保存了：

* `self.query`：问题文本
* `self.options`：带字母的选项字符串
* `self.answer`：正确选项字母
* `self.refrain_label`：拒答选项字母
* `self.protocol`：如果数据表里存在 `protocol` 列就保存，否则为 `None`。([GitHub][5])

方法实现上，它和 `hle.py` 的风格基本一致，但更完整：

* `get_example(index=None)`：随机或按索引取一道题；如果 `dataset == "ProtocolQA"` 且 `protocol` 存在，就把 protocol 也放进 prompt 模板，否则只放 question 和 options。
* `get_iterator()`：顺序遍历全部样本。
* `evaluate(response)`：返回 `accuracy`、`coverage`、`refrain_ratio`、`precision`。
* `output_class()`：返回只包含 `choice` 字段的 Pydantic `MultipleChoiceOutput`。([GitHub][5])

这里还有一个小细节：初始化参数只允许 `DbQA` 或 `SeqQA`，但 `get_example()` 里又判断了 `self.dataset == "ProtocolQA"`。按当前文件逻辑，这个分支实际上不会被触发，因为 `ProtocolQA` 不在允许值里。所以这很像是作者预留了第三类数据集支持，但还没把初始化校验同步改完。就当前代码而言，这也是一个明显的“不一致点”。([GitHub][5])

所以，`lab_bench.py` 的核心功能是：
**把 Lab Bench 类型的生物学多选 benchmark 封装成任务对象，支持随机打乱选项、插入拒答选项、严格格式化回答要求，以及基于拒答机制的评估指标。** ([GitHub][5])

### 这几个文件怎么配合

把这个目录放在一起看，设计思路很清楚：

* `base_task.py` 提供统一接口。
* `hle.py` 和 `lab_bench.py` 分别实现不同 benchmark 的具体任务类。
* 上层 agent 或评测逻辑可以不用关心底层数据集格式，只需要调用统一的方法，比如 `get_example()`、`get_iterator()`、`evaluate()`、`output_class()`。([GitHub][3])

这意味着 `biomni/task` 更像是 Biomni 的 **benchmark task adapter 层**：
把不同数据集包装成统一可调用的任务对象，方便 LLM agent 做题、输出结构化答案、再统一评测。([GitHub][3])

### 一句话总结

* `__init__.py`：空文件，主要用于把目录声明为包。([GitHub][2])
* `base_task.py`：统一任务接口模板。([GitHub][3])
* `hle.py`：Humanity’s Last Exam 生物/医学多选任务封装，支持取题、遍历、评估和输出结构定义，但当前实现里 `refrain_label` 似乎未定义。([GitHub][4])
* `lab_bench.py`：Lab Bench 多选任务封装，支持自动打乱选项、加入“信息不足”拒答项、严格回答格式和基于拒答的评估。([GitHub][5])
