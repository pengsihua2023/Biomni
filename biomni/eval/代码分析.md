这个 `biomni/eval` 文件夹目前很小，只包含 **2 个 Python 文件**：`__init__.py` 和 `biomni_eval1.py`。从代码内容看，它实现的不是训练流程，而是一个 **BiomniEval1 基准数据集的评测加载器与打分器**。也就是说，这一层的职责是：读取标准 benchmark 数据集，按任务类型取样本、查看 prompt/answer，并对用户答案打分。([GitHub][1])

先看整体结构：`__init__.py` 只把 `BiomniEval1` 从 `biomni_eval1.py` 导出，说明这个子包的对外入口就是这个评测类。`biomni_eval1.py` 则定义了 `BiomniEval1` 类和一个 `main()` 演示函数；核心能力包括加载 parquet 数据集、按 `(task_name, task_instance_id)` 建索引、逐题评分、批量评分、列任务、查样本和统计各任务 train/val 数量。([GitHub][2])

`__init__.py` 的功能非常简单：
它执行 `from .biomni_eval1 import BiomniEval1`，并设置 `__all__ = ["BiomniEval1"]`。这意味着外部代码可以直接写 `from biomni.eval import BiomniEval1`，而不用关心具体文件名。它本质上是这个评测子包的“导出入口文件”。([GitHub][2])

`biomni_eval1.py` 是真正的实现文件。它开头的文档字符串已经说明了用途：`BiomniEval1: Evaluation loader for Biomni tasks`，并给出典型用法，例如创建 evaluator 后调用 `evaluate('gwas_causal_gene_opentargets', 0, 'BRCA1')` 进行评分。类初始化时并不要求用户传本地路径，而是直接用 `pandas.read_parquet("hf://datasets/biomni/Eval1/biomni_eval1_dataset.parquet")` 从 Hugging Face 数据集地址读取评测数据。这说明它把 benchmark 数据和评测代码分离了：数据在远程 parquet，逻辑在本地类里。([GitHub][3])

这个类初始化后的第一项关键工作，是建立一个 **实例索引映射**。`__init__` 会遍历整张数据表，把 `(row["task_name"], row["task_instance_id"])` 映射到 DataFrame 的行号 `idx`。这样后续评测时，不需要扫描整个表，只要通过任务名和该任务内部的实例 id 就能快速找到 ground truth。初始化完成后，它还会打印数据集总实例数和任务总数。也就是说，这个类既是数据加载器，也是一个轻量级的 benchmark 索引器。([GitHub][3])

最核心的方法是 `evaluate(task_name, task_instance_id, user_answer)`。这个函数先根据 `(task_name, task_instance_id)` 从索引里找到对应样本，再取出该行的标准答案 `row["answer"]`，然后交给 `_compute_reward(...)` 用任务特定逻辑评分，最后返回 `0.0` 到 `1.0` 之间的分数。如果索引找不到对应实例，会直接报 `ValueError`；如果任务评分逻辑执行时出错，会包成 `RuntimeError` 抛出。换句话说，它实现的是 **“统一接口 + 按任务分流的判分逻辑”**。([GitHub][3])

`_compute_reward(...)` 是整个文件最有信息量的部分，因为它揭示了这个 benchmark 里有哪些任务、以及每种任务如何判分：
`crispr_delivery`、`hle`、以及所有 `lab_bench*` 任务都按 **字母选项大小写不敏感精确匹配**；`gwas_causal_gene*` 和 `screen_gene_retrieval` 按 **基因符号大小写不敏感精确匹配**；`gwas_variant_prioritization` 按 **变异字符串精确匹配**；`rare_disease_diagnosis` 期待用户答案是 JSON 或可解析字典，然后比较其中的 `OMIM_ID` 是否一致；`patient_gene_detection` 则要求用户答案里有 `causal_gene` 字段，如果其中任一预测基因与标准答案中的一个或多个真值基因有交集，就给 1.0，否则 0.0。未知任务名会直接报错。可见这个评测器不是通用 NLP 指标，而是 **按生物医学任务手工定义 reward 规则**。([GitHub][3])

除了评分之外，这个文件还实现了几组很实用的“benchmark 浏览”功能。`get_instance(task_name, task_instance_id)` 会返回一个字典，里面包括全局实例 id、任务内实例 id、任务名、split、prompt 和 answer，便于调试具体题目。`list_tasks()` 返回当前 benchmark 中所有可用任务名。`get_task_stats(task_name=None)` 可以统计某个任务或全部任务的总样本数、train 样本数和 val 样本数；如果不传任务名，还会递归给出每个任务自己的统计信息。所以它不仅能打分，也能当成 **任务清单和数据概览工具** 来用。([GitHub][3])

这个文件还提供了 `batch_evaluate(evaluations)`，支持一次性输入多个 `(task_name, task_instance_id, user_answer)` 三元组并逐个打分；某个样本出错时，它会打印错误并给该项记 `0.0`，而不是让整批中断。这说明作者考虑了批量评测脚本的实际需求。另一个辅助方法 `get_instances_by_task(task_name, split=None)` 则返回某个任务在全部或某个 split 上的 DataFrame 拷贝，方便做下游分析或抽样。再加上 `__repr__` 和 `__len__`，这个类已经具备了一个小型 benchmark SDK 的基本形态。([GitHub][3])

文件末尾的 `main()` 主要是 **演示与自检**。它会实例化 `BiomniEval1`，打印全部任务列表和总体统计，再对 `gwas_variant_prioritization` 的第一条样本展示一次“正确答案得 1.0、错误答案得 0.0”的示例，并演示 `batch_evaluate(...)`。这表明该文件既可作为库被导入，也可直接运行做 smoke test。([GitHub][3])

所以，按“各个文件的功能”来总结：
`__init__.py` 负责把 `BiomniEval1` 暴露成 `biomni.eval` 子包的公共接口；`biomni_eval1.py` 负责实现整个 BiomniEval1 benchmark 的加载、索引、单题评分、批量评分、样本读取和统计汇总。整个 `biomni/eval` 目录当前并没有复杂的多文件评测框架，而是一个相对集中的、单类封装的评测模块。([GitHub][2])
