这个 `biomni/tool` 文件夹可以分成四层来看：

第一层是**实际工具实现文件**，也就是按学科拆分的 `.py` 文件，比如 `biochemistry.py`、`genomics.py`、`database.py`、`literature.py` 等；第二层是**工具注册层**，主要是 `tool_registry.py`；第三层是**包入口/导出层**，主要是 `__init__.py`；第四层是**辅助资源与示例目录**，包括 `example_mcp_tools/`、`protocols/`、`schema_db/`、`tool_description/`。目录页直接列出了这些文件和子目录。([GitHub][1])

先说最关键的两个“框架文件”。

`__init__.py` 很简单，只从 `biomni.utils` 里导出了 `get_tool_decorated_functions`。这说明 `biomni.tool` 这个包本身更像一个“工具集合入口”，方便上层统一发现或加载被装饰过的工具函数。([GitHub][2])

`tool_registry.py` 是这个目录里最明显的“管理中枢”。它定义了 `ToolRegistry` 类，用来接收各模块工具、校验工具格式、分配递增 `id`、按名字或 id 查询工具、列出工具、删除工具，以及把整个 registry 用 `pickle` 保存/加载。它还会把所有工具整理成一个 `document_df`（包含 `docid` 和 `document_content` 两列），明显是在为检索、路由或文档化做准备。工具对象最基本要求包含 `name`、`description`、`required_parameters` 这三个键。([GitHub][3])

再说这一层大量按学科命名的 `.py` 文件。它们不是 description 元数据，而是**真正带函数实现的工具模块**。从代表性文件看，它们普遍有一个共同风格：函数以“单任务工具”为单位，接收结构化参数，执行分析或查询，把结果写文件或返回一段“research log / steps log”式文本。比如 `biochemistry.py` 中的 `analyze_circular_dichroism_spectra` 会分析 CD 光谱、判断二级结构、可选估算热稳定性并把分析文件写到输出目录；同文件里的 `analyze_rna_secondary_structure_features` 会从 dot-bracket 结构提取 stem、loop、配对比例和简化能量特征；后面还有蛋白酶动力学分析函数。([GitHub][4])

所以，按功能可以把这些学科文件理解为 Biomni 的**领域工具库**。目录里这些文件分别对应不同学科：`biochemistry.py`、`bioengineering.py`、`bioimaging.py`、`biophysics.py`、`cancer_biology.py`、`cell_biology.py`、`database.py`、`genetics.py`、`genomics.py`、`glycoengineering.py`、`immunology.py`、`lab_automation.py`、`literature.py`、`microbiology.py`、`molecular_biology.py`、`pathology.py`、`pharmacology.py`、`physiology.py`、`protocols.py`、`support_tools.py`、`synthetic_biology.py`、`systems_biology.py`。从目录命名就能看出这是“按生物医学学科分模块”的设计。([GitHub][1])

其中有几类特别值得单独说。

`database.py` 更像**外部数据库与 API 访问层**，不是单纯做数值分析。它实现了若干通用底座函数：`get_hpo_names` 用本地 `hp.obo` 解析 HPO 术语名称；`_query_llm_for_api` 会调用 Biomni 的统一 LLM 接口，把自然语言 prompt 和 schema 交给模型生成结构化 API 调用参数；`_query_rest_api` 统一封装了 GET/POST 请求、JSON 解析和错误处理；`_query_ncbi_database` 则是针对 NCBI 数据库查询的核心入口。它导入了 `requests`、Biopython 的 BLAST 模块、`langchain_core.messages`，以及 `biomni.llm.get_llm`，这说明这个模块负责把“自然语言问题 → API/数据库调用 → 结果返回”串起来。([GitHub][5])

`literature.py` 是**文献检索与文献抓取层**。从已打开的代码看，它至少实现了：按 DOI 去解析出版商页面并下载 supplementary material 的 `fetch_supplementary_info_from_doi`；通过 `arxiv` 包查 arXiv 的 `query_arxiv`；通过 `scholarly` 查 Google Scholar 的 `query_scholar`；以及 PubMed 查询函数 `query_pubmed`。它导入了 `requests`、`BeautifulSoup`、`PyPDF2`、`googlesearch` 等，说明这个模块偏“网页抓取 + 学术搜索 + PDF/补充材料提取”。([GitHub][6])

`protocols.py` 则是**实验 protocol 检索层**。它直接对接 protocols.io API，读取环境变量或配置里的 access token，提供 `search_protocols` 和 `get_protocol_details` 两类能力：前者按关键词搜公开 protocol，后者根据 protocol id 拉取详情。这和 `literature.py` 一样，属于“知识/文档检索型工具”，但面向的是实验步骤而不是论文。([GitHub][7])

`support_tools.py` 是**通用辅助工具层**。它提供了一个持久化命名空间的 `run_python_repl`，允许多次执行 Python 命令并保留变量；还提供 Matplotlib 图像捕获辅助函数、`read_function_source_code` 用于读取任意模块函数源码、以及 `download_synapse_data` 用来用 Synapse CLI 下载数据。这个模块不绑定具体学科，而是给 agent 执行代码、读源码、下载数据。([GitHub][8])

`genomics.py` 和 `systems_biology.py` 代表了另外两种典型实现风格。`genomics.py` 里可以看到跨物种基因转换 `interspecies_gene_conversion`、ARCHS4 RNA-seq 表达查询 `get_rna_seq_archs4`、基因集富集分析 `gene_set_enrichment_analysis`、Hi-C 染色质互作分析 `analyze_chromatin_interactions` 等函数，说明它是偏组学分析与单细胞/染色质分析的“大工具箱”。`systems_biology.py` 则包含 `query_chatnt`（用 DNA 模型回答序列问题）、`perform_flux_balance_analysis`、`model_protein_dimerization_network`、`simulate_metabolic_network_perturbation` 等，明显偏系统生物学建模、网络动力学和代谢模拟。([GitHub][9])

综合起来，这个目录下各代码文件的功能可以这样理解：

`biochemistry.py` 到 `systems_biology.py` 这一长串学科文件，是 Biomni 的**核心工具实现层**。它们把各类生物医学分析、数据库访问、文献检索、建模计算、图像处理、实验自动化等能力封装成一个个独立函数。目录结构本身就表明 Biomni 想把不同学科任务拆成可调用工具模块。([GitHub][1])

`tool_registry.py` 是这些工具的**统一索引和注册器**，把各模块的工具汇总成一个统一 registry，并提供 id/name 检索和持久化。([GitHub][3])

`__init__.py` 是**轻量入口文件**，用于暴露工具发现相关函数。([GitHub][2])

`example_mcp_tools/` 是**MCP 示例工具目录**，从名字看是给 MCP 集成或演示用；`tool_description/` 是你前面问过的**工具说明元数据层**；`schema_db/` 从命名看是 schema/数据库相关资源目录；`protocols/` 是 protocol 相关资源目录。它们都在同一个 `tool` 目录下，说明 Biomni 把“实现、注册、说明、示例、资源”放在同一工具子系统里。([GitHub][1])
