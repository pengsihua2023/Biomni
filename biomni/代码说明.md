这个 `biomni/` 文件夹是整个 Biomni 包的**核心源码根目录**。从目录结构看，它包含 6 个主要子系统目录：

* `agent/`
* `biorxiv_scripts/`
* `eval/`
* `know_how/`
* `model/`
* `task/`
* `tool/`

以及 7 个根层代码文件：

* `__init__.py`
* `config.py`
* `env_desc.py`
* `env_desc_cm.py`
* `llm.py`
* `utils.py`
* `version.py` ([GitHub][1])

如果只看这个根目录，功能可以分成两层：

一层是**子系统目录**，分别承载 agent、工具、模型、评测、任务和论文脚本等大模块；另一层是**根层支撑文件**，负责包入口、全局配置、数据/库描述、LLM 适配、通用工具和版本号。([GitHub][1])

下面按文件说明。

## 1. `__init__.py`

这个文件非常简单，只做了一件事：从 `version.py` 导入 `__version__`，并把它作为包公开接口导出。也就是说，外部代码可以直接通过 `biomni.__version__` 读取版本号。([GitHub][2])

## 2. `config.py`

这个文件实现了 Biomni 的**集中配置管理**。核心是一个 `BiomniConfig` 数据类，统一管理这些设置：

* 数据路径 `path`
* 超时时间 `timeout_seconds`
* 默认模型 `llm`
* 温度 `temperature`
* 是否启用工具检索器 `use_tool_retriever`
* 是否启用商业模式 `commercial_mode`
* 自定义模型服务的 `base_url` 和 `api_key`
* 模型来源 `source`
* protocols.io token `protocols_io_access_token` ([GitHub][3])

它还会在 `__post_init__` 里自动读取环境变量覆盖默认值，例如：

* `BIOMNI_DATA_PATH`
* `BIOMNI_TIMEOUT_SECONDS`
* `BIOMNI_LLM`
* `BIOMNI_COMMERCIAL_MODE`
* `BIOMNI_CUSTOM_BASE_URL`
* `PROTOCOLS_IO_ACCESS_TOKEN` 等。([GitHub][3])

所以 `config.py` 的作用是：**把 Biomni 的运行参数统一收口，并保持和 `.env`/环境变量兼容。** ([GitHub][3])

## 3. `env_desc.py`

这个文件实现的是 Biomni 的**环境资源说明表**，主要包含两个大字典：

* `data_lake_dict`
* `library_content_dict` ([GitHub][4])

其中 `data_lake_dict` 用“文件名 → 描述”的方式列出了大量 Biomni 数据湖资源，比如：

* DepMap 数据
* GTEx 表达数据
* OMIM
* HPO
* MSigDB 基因集
* Drug interaction 数据
* 单细胞/蛋白互作/病毒宿主互作等资源。([GitHub][4])

`library_content_dict` 则是“软件库/包名 → 描述”的字典，覆盖很多 Python 和 R 包，比如：

* `scanpy`
* `anndata`
* `rdkit`
* `gseapy`
* `pandas`
* `numpy`
* `scikit-learn`
* `PyPDF2`
* `arxiv`
* `scholarly`
* `cobra` 等。([GitHub][4])

所以 `env_desc.py` 的作用是：**告诉 agent 当前有哪些数据资源和软件库可用，并给出自然语言描述，便于检索和路由。** ([GitHub][4])

## 4. `env_desc_cm.py`

这个文件和 `env_desc.py` 是同类文件，但它对应的是**商业模式（commercial mode）下可用的资源说明**。从 `a1.py` 的设计思路和文件命名看，普通模式和商业模式使用不同的环境描述文件；`env_desc_cm.py` 是商业模式那一套。当前可见内容里，它同样包含库和资源描述条目。([GitHub][1])

所以它的作用可以概括为：**在商业模式下提供受许可范围约束的资源/库描述集合。** 这个判断和它的文件名、目录位置以及 `commercial_mode` 配置项是对应的。([GitHub][3])

## 5. `llm.py`

这个文件实现的是 Biomni 的**统一 LLM 适配层**。核心函数是 `get_llm(...)`，它会根据模型名、来源、base URL、API key 和配置对象，创建合适的聊天模型实例。([GitHub][5])

它支持的来源包括：

* `OpenAI`
* `AzureOpenAI`
* `Anthropic`
* `Ollama`
* `Gemini`
* `Bedrock`
* `Groq`
* `Custom` ([GitHub][5])

它还有一套**自动识别模型来源**的逻辑，例如：

* `claude-*` → Anthropic
* `gpt-*` → OpenAI
* `azure-*` → Azure OpenAI
* `gemini-*` → Gemini
* 含 `llama`、`mistral`、`qwen` 等名字时倾向 Ollama
* Bedrock 风格前缀时走 Bedrock。([GitHub][5])

另外，它对 `gpt-5` 这类新模型还有专门兼容处理，会强制使用 Responses API，并去掉不兼容的 `stop` 和 `temperature` 参数，避免 400 错误。([GitHub][5])

所以 `llm.py` 的功能就是：**屏蔽不同大模型供应商的差异，为上层 agent 提供一个统一的模型创建接口。** ([GitHub][5])

## 6. `utils.py`

这个文件实现的是 Biomni 的**通用工具函数库**。当前可见内容里，比较明确的功能有：

* `run_r_code(code)`：把 R 代码写入临时文件并用 `Rscript` 执行
* `run_bash_script(script)`：把 Bash 脚本写入临时 `.sh` 并执行
* `run_cli_command(command)`：直接运行命令行
* `run_with_timeout(...)`：带超时执行函数。([GitHub][6])

它还导入了很多和工具包装、验证、交互环境、数据下载相关的模块，例如：

* `StructuredTool`
* `BaseCallbackHandler`
* `ValidationError`
* `requests`
* `pandas`
* `pickle`
* `zipfile` 等。([GitHub][6])

所以 `utils.py` 不是某个单一功能，而是 Biomni 的**杂项基础设施层**：负责脚本执行、命令调用、超时控制，以及一批底层辅助函数。([GitHub][6])

## 7. `version.py`

这个文件只负责定义 Biomni 的版本号，也就是包级别的 `__version__` 来源。它配合 `__init__.py` 和打包配置一起使用。([GitHub][7])

---

## 子目录在根层中的定位

虽然你这次问的是 `biomni/` 根目录，但从目录名就能看出这些子目录是主要功能分区：

* `agent/`：代理层
* `tool/`：工具实现层
* `model/`：模型/检索相关层
* `task/`：任务定义层
* `eval/`：评测层
* `biorxiv_scripts/`：从 bioRxiv 论文中抽任务和生成函数的脚本
* `know_how/`：经验文档或知识资源层。([GitHub][1])

所以整个 `biomni/` 根目录本质上是一个**科研 agent 平台的总装配包**：根层文件负责全局配置和基础设施，子目录负责具体子系统实现。([GitHub][1])

一句话总结：

* `__init__.py`：导出版本号
* `config.py`：集中配置管理
* `env_desc.py`：普通模式的资源/库描述字典
* `env_desc_cm.py`：商业模式的资源/库描述字典
* `llm.py`：统一 LLM 适配与模型创建
* `utils.py`：通用执行与辅助函数
* `version.py`：版本号定义
* 其余子目录则承载 agent、tool、model、task、eval 等主要功能模块。 ([GitHub][2])

下面是 `biomni/` 根目录的对照表版总结：

| 文件/目录              | 作用                           | 典型调用方                                     |
| ------------------ | ---------------------------- | ----------------------------------------- |
| `__init__.py`      | 导出包版本号 `__version__`         | 外部导入 `biomni` 时读取版本信息                     |
| `config.py`        | 统一管理 Biomni 运行配置，支持环境变量覆盖    | `A1`、LLM 初始化、protocol/API 配置读取            |
| `env_desc.py`      | 普通模式下的数据湖与软件库说明字典            | `A1` 做资源检索、工具路由时使用                        |
| `env_desc_cm.py`   | 商业模式下的资源与库说明字典               | `A1(commercial_mode=True)`                |
| `llm.py`           | 统一创建不同来源的大模型客户端              | `A1`、`qa_llm`、`FunctionGenerator`、数据库查询模块 |
| `utils.py`         | 通用辅助函数，如执行 R、bash、CLI、超时控制等  | 工具层、agent 层、脚本执行相关模块                      |
| `version.py`       | 定义 Biomni 版本号                | `__init__.py`、打包配置 `pyproject.toml`       |
| `agent/`           | 主代理与子代理实现                    | 用户直接调用 `A1`，以及教程 notebook                 |
| `tool/`            | 各学科工具实现、注册、schema、protocol 等 | `A1`、MCP server、工具检索器                     |
| `model/`           | 模型相关组件，例如资源检索器               | `A1` 的 prompt-based retrieval             |
| `task/`            | 任务抽象与 benchmark task adapter | HLE、Lab Bench 等任务执行与评估                    |
| `eval/`            | Benchmark 评测加载与打分            | `BiomniEval1` 评测流程                        |
| `biorxiv_scripts/` | 从 bioRxiv 论文抽任务并生成函数的脚本      | 方法学挖掘、任务生成流水线                             |
| `know_how/`        | 经验文档/协议/最佳实践知识资源             | `A1` 的 know-how 检索与引用                     |

再用一句更容易记的话来概括：

`biomni/` 根目录像一个总装配层——
`config.py` 和 `llm.py` 管“怎么运行”，`env_desc*.py` 管“有什么资源”，`utils.py` 管“底层怎么执行”，而 `agent/ tool/ model/ task/ eval/` 这些子目录负责真正的科研 agent 能力。

如果你继续往下看，最适合的下一步是把 `A1` 这条主线串起来：
`config.py → llm.py → env_desc.py → tool/ → model/retriever.py → agent/a1.py`。
